Framework D.N.E.E. Docs

Um processo leve para documentar, evoluir e manter projetos com apoio de IA.

O Conceito A Estrutura As 4 Abas Como Aplicar Dúvidas da IA Dia a Dia

O Problema & A Solução

Documentação viva para projetos feitos com IA.

A IA acelera a criação de código, mas sem documentação viva ela também pode gerar perda de contexto, decisões soltas e retrabalho. O D.N.E.E. Docs cria um fluxo simples para descobrir o problema, nortear o projeto, planejar mudanças e registrar evidências.

1. Descobrir

A IA avalia o problema, levanta a hipótese de valor e propõe a solução no chat antes de tocar no código.

2. Nortear

A IA lê o Norte do projeto (SDD.md) para garantir que a solução proposta respeita as regras e a arquitetura vigentes.

3. Especificar

A IA transforma a solução aprovada em tarefas claras no ROADMAP.md, critérios de aceite no SDD.md e próximos passos controlados antes de gerar código.

4. Evidenciar

Após sucesso, a IA documenta automaticamente o Antes, Depois e a Decisão no CHANGELOG_EVIDENCES.md.

A Estrutura de Arquivos
(simples e padronizada)

A regra de ouro: a fonte da verdade são os arquivos Markdown dentro de docs/. A página visao-projeto.html é apenas a interface visual para leitura humana. Nada além disso é obrigatório.

.cursorrules
A Constituição (O Maestro)

Arquivo raiz que obriga a IA a seguir o processo D.N.E.E. Docs, pensar como PM e manter a documentação sempre atualizada.

docs/visao-projeto.html
A Interface Visual

Um HTML que renderiza os arquivos Markdown num painel de 4 abas. Feito para humanos (gestores, QA, time). Não é fonte da verdade: só exibe.

docs/SOBRE.md
O Pitch (A Porta de Entrada)

A visão de negócio: problema, público e valor em linguagem leiga. O Discovery resumido também mora aqui. É a aba padrão.

docs/SDD.md
O Norte Técnico (O Presente)

A verdade técnica do projeto: escopo, fluxos, arquitetura, regras importantes e critérios de qualidade e aceite.

docs/ROADMAP.md
O Futuro (O Planejamento)

Fases, backlog priorizado, tarefas, pendências e débitos técnicos. É daqui que a IA consome as próximas ações.

docs/CHANGELOG_EVIDENCES.md
O Passado (Decisões & Evidências)

Histórico de decisões e valor gerado. Formato estrito com hora: AAAA-MM-DD · HH:MM — Mudança.

meu-projeto/

.cursorrules <- Regras da IA

docs/

visao-projeto.html <- Interface visual

SOBRE.md <- Porta de Entrada

SDD.md <- O Norte Técnico

ROADMAP.md <- O Futuro

CHANGELOG_EVIDENCES.md <- O Passado

src/

... seu código limpo aqui

Camada opcional: para projetos muito grandes, com múltiplos devs, regras críticas ou necessidade de specs formais, o time pode adicionar uma pasta extra de especificações. No fluxo padrão, porém, o D.N.E.E. Docs funciona apenas com a pasta docs/.

As 4 Abas da Documentação Visual

Todo projeto documentado com D.N.E.E. Docs tem uma página docs/visao-projeto.html com exatamente 4 abas. Nada de abas extras como Discovery ou Specs: cada assunto tem um lugar fixo.

💡 Sobre o projeto

Renderiza o SOBRE.md. Visão de negócio, público, valor e o Discovery resumido. É a aba padrão ao abrir.

🧭 Norte do projeto

Renderiza o SDD.md. Critérios, regras, fluxos e estrutura técnica ficam todos aqui.

🗺️ Roadmap

Renderiza o ROADMAP.md. Planejamento, tarefas, backlog e pendências.

📌 Decisões & Evidências

Renderiza o CHANGELOG_EVIDENCES.md. Histórico, decisões e evidências de entrega.

Lembre: os .md são a fonte da verdade. A página visual só exibe. Sempre atualize primeiro o Markdown e depois a página, se necessário.

Como Aplicar na Prática

Em um projeto do zero, o objetivo é nascer com a governança correta: Discovery primeiro, arquivos depois.

  • 1

    Abra o chat do Cursor no seu repositório limpo.

  • 2

    Copie o Prompt de Projeto Novo (abaixo) e envie para a IA.

  • 3

    A IA faz o mini-Discovery com você no chat e aguarda seu OK antes de criar qualquer arquivo.

  • 4

    Inicie o desenvolvimento. Daqui pra frente a IA já está instruída a planejar antes de codar e registrar evidências sozinha.

🌱 Prompt de Projeto Novo (Copiar & Colar)

Cursor, vamos iniciar um projeto novo usando o padrão D.N.E.E. Docs.

Meu objetivo com este projeto é:
[DESCREVA SUA IDEIA AQUI]

Antes de criar arquivos ou código, faça um mini-Discovery comigo no chat.

Responda primeiro:

1. Qual problema este projeto resolve?
2. Quem será impactado ou beneficiado?
3. Qual é a hipótese de valor?
4. Qual solução inicial faz sentido?
5. Quais seriam os critérios de sucesso?
6. Quais informações ainda precisam ser validadas?

Use a marcação [A VALIDAR] para tudo que não estiver claro.
Não invente informações para parecer completo.
Ao final, me faça perguntas objetivas para complementar o projeto.

Aguarde meu OK antes de criar arquivos.

Após meu OK, crie a estrutura obrigatória:

3. Sempre atualize primeiro os .md e depois a página visual, se necessário.
4. A página visual deve ter exatamente 4 abas:
   - 💡 Sobre o projeto
   - 🧭 Norte do projeto
   - 🗺️ Roadmap
   - 📌 Decisões & Evidências
5. A aba "💡 Sobre o projeto" deve abrir por padrão.
6. A página deve ter layout padronizado, limpo, responsivo e independente da identidade visual da ferramenta.
7. Não copie o layout do produto. Use sempre o padrão D.N.E.E. Docs.
8. Se não souber algo, registre como [A VALIDAR] e crie uma seção "Pontos a complementar".
9. Inicie o CHANGELOG_EVIDENCES.md com o registro da criação da documentação.

    O Raio-X (Mapeamento Atual)

    A IA varre o código e entrega no chat o estado real: funcionalidades, fluxos, stack, dados, regras percebidas e pontos frágeis — marcando o que for incerto.

    A Criação da Documentação

    As Regras de Continuidade

    O .cursorrules passa a obrigar o fluxo D.N.E.E. Docs em toda mudança relevante: ler os docs, fazer Discovery, aguardar aprovação, implementar e registrar evidência.

PASSO 3 — Regras para continuidade

Atualize o .cursorrules para obrigar que, daqui em diante, toda mudança relevante siga este fluxo:

1. Ler docs/SOBRE.md
2. Ler docs/SDD.md
3. Ler docs/ROADMAP.md
4. Ler docs/CHANGELOG_EVIDENCES.md
5. Fazer Discovery curto no chat
6. Aguardar aprovação
7. Implementar
8. Atualizar o Markdown correspondente
9. Atualizar docs/visao-projeto.html apenas se necessário
10. Registrar decisão/evidência em CHANGELOG_EVIDENCES.md

Nunca implementar mudança relevante sem atualizar a documentação relacionada.

No final, me entregue:
1. Arquivos criados ou alterados
2. Resumo do estado atual documentado
3. Principais débitos encontrados
4. Pontos marcados como [A VALIDAR]
5. Perguntas para complementar
6. Próximas ações recomendadas

📚 O que fazer quando a IA não souber algo?

A IA não deve preencher lacunas inventando informações. Quando algo não estiver claro, ela deve registrar a dúvida e seguir com uma marcação padronizada.

[A VALIDAR]

Quando a informação precisa ser confirmada.

[PERGUNTA AO RESPONSÁVEL]

Quando a IA precisa de uma resposta humana para continuar.

[HIPÓTESE — CONFIRMAR]

Quando existe uma hipótese provável, mas ela ainda não foi validada.

Todo arquivo Markdown deve terminar com uma seção "Pontos a complementar". Assim, a documentação nunca fica falsa: ela fica viva, rastreável e aberta para evolução.

🔄 Rotina & Manutenção (O Dia a Dia)

Depois que o projeto está estruturado, você usará os prompts abaixo no dia a dia para consumir tarefas e manter a IA na rédea curta.

A. Executando o Roadmap 🚀

Como consumir as pendências e gerar código de forma segura.

Cursor, vamos executar a próxima tarefa do projeto seguindo o padrão D.N.E.E. Docs.

Antes de mexer no código, leia:

- docs/SOBRE.md
- docs/SDD.md
- docs/ROADMAP.md
- docs/CHANGELOG_EVIDENCES.md

Depois, escolha a próxima tarefa pendente mais relevante do ROADMAP.md e faça um Discovery curto no chat:

1. Qual problema esta tarefa resolve?
2. Qual parte do projeto será impactada?
3. Qual solução você propõe?
4. Quais arquivos provavelmente serão alterados?
5. Quais riscos existem?
6. Como vamos validar?
7. Qual evidência deve ser registrada ao final?

Aguarde minha aprovação antes de implementar.

Depois da implementação:
- Atualize o ROADMAP.md marcando a tarefa como concluída, se aplicável.
- Atualize o SDD.md se alguma regra, fluxo ou arquitetura mudou.
- Atualize o SOBRE.md se a funcionalidade mudou a percepção do produto.
- Atualize o CHANGELOG_EVIDENCES.md com Antes, Depois, Decisão, Evidência e Próximos passos.
- Atualize o docs/visao-projeto.html apenas se necessário.

B. Correção de Rota (O "Chicote") ⚡

Use quando a IA codar certo, mas "esquecer" da documentação.

Cursor, você implementou uma mudança, mas ainda falta atualizar a documentação conforme o padrão D.N.E.E. Docs.

Revise a alteração feita e atualize obrigatoriamente:

1. docs/ROADMAP.md
   - Marque como concluído o item relacionado, se existir.
   - Adicione pendências novas, se surgiram.

2. docs/SDD.md
   - Atualize regras, fluxos, dados, arquitetura ou critérios de qualidade se a mudança impactou algum desses pontos.

3. docs/SOBRE.md
   - Atualize apenas se a mudança alterou a visão, público, valor, funcionalidades ou momento atual do projeto.

4. docs/CHANGELOG_EVIDENCES.md
   - Registre a entrega no formato:

## AAAA-MM-DD · HH:MM — [Nome da mudança]

### Antes
### Depois
### Decisão
### Evidência
### Impacto observado
### Próximos passos

5. docs/visao-projeto.html
   - Atualize apenas se houve mudança de layout, abas, renderização ou conteúdo visual estático.

Não invente métricas.
Se não houver evidência real, registre como [A VALIDAR] e indique qual evidência precisa ser coletada.
